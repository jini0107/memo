[팁 1. God Object(신 객체) 안티패턴 피하기]
현재 App.tsx 파일이 800줄이 넘어가며 라우팅, 비즈니스 로직, 상태 관리, UI 구성 등 "너무 많은 일"을 하고 있습니다.
이를 소프트웨어 공학에서는 'God Object'라고 부릅니다. 전능하지만 조금만 건드려도 사이드 이펙트가 발생하기 쉽죠.
해결책: 
- 커스텀 훅 분리 (예: useItems, usePinModal, useImageHandler)
- 페이지(혹은 뷰) 단위의 컴포넌트 분리 (예: DashboardView, ItemDetailView)

[팁 2. 오프라인 퍼스트(Offline-First) 아키텍처]
모바일 환경에서는 인터넷이 끊길 수 있습니다. 현재 LocalStorage를 쓰지만 이미지 용량 때문에 한계에 부딪히고 있습니다.
해결책:
- IndexedDB (localForage 라이브러리 추천) 를 사용하여 기기 용량 한계까지 로컬에 저장
- 브라우저의 navigator.onLine 이벤트와 ServiceWorker를 활용하여, 오프라인일 때 로컬에 저장하고 온라인이 되면 Supabase와 백그라운드 동기화 수행

[팁 3. 보안 정보의 생명주기 관리]
클라이언트(브라우저)에서 사용하는 PIN 번호는 절대로 '평문(Plain Text)'으로 저장해서는 안 됩니다.
해결책:
- 입력받은 PIN 번호를 Web Crypto API를 사용하여 SHA-256 해시로 변환 후 저장
- 인증 시에도 입력받은 값을 해시하여 저장된 해시값과 비교 (원문은 코드 어디에도 남지 않게 됨)

[팁 4. 벤더 종속성(Vendor Lock-in)과 저장소 추상화 전략]
초기 개발 시 특정 클라우드 서비스(Supabase, Firebase 등)에 데이터를 저장하다가 다른 서비스로 갈아타게 될 때를 대비해야 합니다.
해결책 (Repository Pattern): 
- 클라우드와 통신하는 코드를 `App.tsx`나 컴포넌트 내부에 직접 하드코딩하지 않습니다.
- 대신 `supabaseService.ts`처럼 별개의 파일(서비스 계층)로 분리하고 `fetchItems()`, `addItem()` 처럼 이름 지어진 함수만 호출하세요.
- 이렇게 "관심사를 분리(Separation of Concerns)" 해두면, 나중에 AWS나 Firebase로 갈아탈 때 그 파일 하나만 교체하면 화면 코드(UI)는 단 한 줄도 수정할 필요가 없게 됩니다.

[팁 5. 서버 비용 최적화(Cost Optimization)와 로컬 지향 아키텍처]
"서버를 거치지 않고 사용자 폰에만 저장하면 안 될까?" 라는 발상은 개발 리더/비즈니스 관점에서 가장 훌륭한 접근입니다.
- 장점 (비용 Zero): 웹 브라우저의 IndexedDB 기술을 사용하여 사진을 클라이언트 측(사용자 디바이스)에서 암호화 후 즉시 로컬에 격리 저장하면, 클라우드 서버 유지 비용이 0원이 되고 비행기 모드에서도 초고속 로딩이 가능합니다.
- 단점 (데이터 휘발성): 사용자가 브라우저 '인터넷 사용 기록/데이터 삭제'를 실행하거나 디바이스를 잃어버리면 소중한 사진들이 영구 증발합니다. 또한 멀티 디바이스(PC, 태블릿) 동기화가 불가능해집니다.
- 절충안 (하이브리드 전략): 가장 용량을 많이 먹는 '사진' 데이터는 폰(로컬 IndexedDB)에만 보관하고, 용량이 가벼운 '물건 이름, 카테고리 등의 메타 텍스트' 정보만 BaaS(Supabase 등)의 평생 무료 요금제(Free Tier) 범위 내에 동기화 백업하는 것이 현실적입니다.

[팁 6. 무료 클라우드(BaaS) 3대장 비교와 선택 기준]
초기 1인 개발 시 서버 개발을 스킵하고 프론트엔드 작업에만 집중하게 도와주는 BaaS(Backend as a Service)의 3대장 비교입니다.
1. Firebase (Google): NoSQL 기반. 생태계 최강, 초보자 접근성 최고. 하지만 관계형 데이터가 얽히기 시작하면 쿼리 복잡도가 폭증하여 유지보수가 지옥이 됨.
2. Supabase (오픈소스): RDBMS(PostgreSQL) 기반. 현재 최고의 트렌드이자 Firebase의 강력한 대항마. 데이터 정합성 관리에 탁월하나 완전 초보에겐 관계형 DB 기초 지식이 필요함.
3. Appwrite: 초보자 친화적이고 직관적인 UI. 급격히 성장 중이나 레퍼런스(한글 자료) 부족이 단점.
▶ 시니어의 멘토링 결론: 복잡한 데이터 조회가 필요 없다면 Firebase가 가장 무난하나, 구조적 성장을 생각한다면 트렌디한 Supabase가 월등한 선택. (현재 앱의 설정은 탁월함!)
