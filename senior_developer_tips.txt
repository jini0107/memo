================================================================================
[WhereIsIt] 프로젝트 성공을 위한 시니어 개발자의 핵심 가이드 & 팁
================================================================================

당신의 10년 차 사수로서, 프로젝트 운영 중 꼭 기억해야 할 핵심 팁들을 정리해 드립니다.
이 내용을 잘 숙지하시면 안전하고 전문적인 개발자가 될 수 있습니다!

1. 보안 및 환경 설정 (Security & Config)
--------------------------------------------------------------------------------
□ .env 파일 관리 (비밀 금고)
   - API 키나 DB 비밀번호는 소스 코드에 직접 적지 말고 반드시 .env에 보관하세요.
   - GitHub 등에 올릴 때 이 파일은 공유되지 않도록 .gitignore에 등록되어 있어야 합니다. (현재 완료)

□ 브라우저 노출 주의 (프론트엔드 보안)
   - 현재 방식(VITE_ 사용)은 브라우저에서 키가 노출될 수 있는 구조입니다. 
   - 아주 중요한 대규모 프로젝트라면 중간에 Backend 서버를 두는 것이 정석입니다.

□ .gitignore의 중요성
   - node_modules, dist, .env 등은 절대로 소스 저장소(Git)에 올리지 않습니다.
   - 프로젝트가 무거워지거나 보안 데이터가 유출되는 것을 막아줍니다.

2. 데이터 관리 및 인프라 (Data & Infra)
--------------------------------------------------------------------------------
□ Supabase 연동 (클라우드 저장소)
   - 앱 새로고침 후에도 데이터가 남으려면 Supabase SQL을 반드시 실행해야 합니다.
   - RLS(Row Level Security) 정책을 통해 내 데이터가 안전한지 항상 체크하세요.
   - 공개 개발 단계가 끝나면 'Public Access' 정책을 'Authenticated'로 변경하는 것이 좋습니다.

□ 이미지 최적화 (모바일 환경 고려)
   - 모바일 기기는 메모리가 제한적이므로 고화질 원본 사진은 앱을 느리게 만듭니다.
   - 현재 우리 앱은 400px 수준으로 자동 압축하도록 되어 있어 매우 효율적입니다.

3. AI 연동 가이드 (AI Integration)
--------------------------------------------------------------------------------
□ Gemini API 활용
   - AI는 질문(Prompt)에 따라 결과가 크게 달라집니다.
   - 한국어 서비스라면 프롬프트도 한국어로 명확하게 작성하면 더 정확한 결과가 나옵니다.
   - (현재 프롬프트는 한국어 최적화 튜닝을 완료해 두었습니다.)

4. 문제 해결 및 디버깅 (Troubleshooting)
--------------------------------------------------------------------------------
□ ERR_CONNECTION_REFUSED 해결
   - 이 에러는 개발 서버(npm run dev)가 꺼져 있을 때 발생합니다. 당황하지 말고 터미널에서 서버를 다시 켜주세요.

□ F12 개발자 도구 활용
   - 네트워킹 오류가 나거나 AI 응답이 안 오면 브라우저의 F12 -> Network 탭을 보세요. 
   - 에러의 진짜 원인(403, 500 등)을 가장 빨리 찾을 수 있는 방법입니다.

5. 시니어의 마인드셋 (Developer Mindset)
--------------------------------------------------------------------------------
□ "동작만 하는 코드"보다 "안전한 코드"
   - 항상 방어적 코딩(Edge case 처리)을 하세요. 
   - 사용자가 이상한 값을 넣거나 인터넷이 끊겼을 때 앱이 뻗지 않게 처리하는 것이 실력입니다.

□ 공식 문서(Docs)와 친해지기
   - 누군가의 블로그 글보다 라이브러리/프레임워크의 공식 사이트를 먼저 믿으세요.

6. 상업용 UI/UX 디자인 핵심 (Premium UI Design)
--------------------------------------------------------------------------------
□ 디자인 시스템 (Design System)
   - CSS 변수(Custom Properties)나 TailwindCSS 커스텀 테마로 색상, 간격, 라운드를 통일하세요.
   - 일관된 톤과 스타일이 "프리미엄한 느낌"의 핵심입니다.
   - 팔레트는 Primary/Accent/Surface 3계열 + Danger/Warning 2계열이 기본입니다.

□ Empty State의 중요성
   - 처음 앱을 켰을 때 빈 화면은 사용자가 "이 앱이 뭐지?" 하고 이탈하는 원인 1위입니다.
   - 히어로 일러스트 + 핵심 기능 소개 + CTA 버튼으로 앱의 가치를 즉시 전달하세요.

□ 마이크로 인터랙션 (Micro-interactions)
   - 버튼 클릭, 카드 터치, 모달 오픈 시 0.15~0.3초의 미세한 애니메이션이 "살아있는 느낌"을 줍니다.
   - CSS animation + transition으로 구현 가능하며, 성능 부담은 거의 없습니다.
   - 단, 과도한 애니메이션은 오히려 사용성을 해칩니다.

□ 글래스모피즘 (Glassmorphism)
   - backdrop-filter: blur() + 반투명 배경으로 고급스러운 레이어 효과를 만들 수 있습니다.
   - 헤더, 모달 오버레이, 네비게이션에 적용하면 효과적입니다.

□ 모바일 우선 설계 (Mobile First)
   - 터치 영역은 최소 44px × 44px을 유지하세요 (Apple HIG 기준).
   - 바텀 시트(Bottom Sheet) 패턴은 모바일에서 가장 자연스러운 모달 방식입니다.
   - Safe Area Inset을 반드시 적용해야 노치/홈 인디케이터와 겹치지 않습니다.

□ 폰트 선택
   - Inter, Pretendard, Noto Sans KR 등 가독성 높은 폰트를 사용하세요.
   - 한국어와 영어가 모두 깔끔하게 보이는 폰트가 중요합니다.
   - font-weight 변화로 정보의 계층 구조(Hierarchy)를 만드세요.

7. 보안 기능 구현 팁 (Security Implementation)
--------------------------------------------------------------------------------
□ 절대적인 보안은 없다
   - 클라이언트 사이드(프론트엔드) 잠금은 기술적으로 우회 가능합니다.
   - 하지만 일반적인 사용자의 실수를 방지하고 프라이버시를 지키는 데는 충분합니다.

□ UX와 트레이드오프 (Trade-off)
   - 보안이 강할수록 불편해집니다 (비밀번호 자주 입력 등).
   - "힌트 보기", "단계별 잠금(1분→1시간)" 등 사용자를 배려하는 장치를 꼭 넣으세요.

8. 로컬 미리보기 및 테스트 (Local Preview)
--------------------------------------------------------------------------------
□ 개발 서버 상시 실행 (Preview)
   - 코드 수정을 실시간으로 확인하려면 `npm run dev` 명령어를 입력하고 표시된 로컬 주소(예: http://localhost:5173)로 접속하세요.
   - Vite의 HMR(Hot Module Replacement) 기능 덕분에 파일을 저장하면 브라우저가 자동으로 새로고침됩니다.

□ 모바일 미리보기
   - PC 브라우저에서 'F12 -> 모바일 기기 아이콘'을 클릭하면 실제 핸드폰 화면 크기에서 어떻게 보이는지 미리 볼 수 있습니다.

9. 지속적 통합 및 자동 배포 (Continuous Deployment)
--------------------------------------------------------------------------------
□ "작은 수정, 빠른 업로드" 원칙
   - 변경 사항이 생길 때마다 즉시 커밋하고 푸시하는 습관은 매우 좋습니다.
   - 그래야 현재처럼 GitHub Pages([https://jini0107.github.io/memo/](https://jini0107.github.io/memo/))를 통해 어디서든 최신 화면을 확인할 수 있습니다.

□ Atomic Commit (원자적 커밋)
   - 하나의 커밋에는 하나의 의미 있는 작업만 담으세요. 
   - 나중에 문제가 생겼을 때 특정 지점으로 되돌리기 훨씬 수월해집니다.
